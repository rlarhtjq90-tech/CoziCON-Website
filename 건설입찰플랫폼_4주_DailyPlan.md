# 건설 입찰 플랫폼 — 4주 일자별 실행 계획

**범위**: MVP (Phase 1~3 핵심 + Phase 4·5 일부)
**전제**: 회원가입/로그인 완료 / 1인 또는 소수 개발자 풀타임 기준
**원칙**:

- 매일 1~3개의 구체적인 산출물(Deliverable)
- 매주 끝나면 배포 가능한 상태로 마감
- 외부 검증/연동(국세청, KISCON, PG)은 신청·심사 기간이 있으니 **Day 1에 즉시 신청**

---

## 4주 큰 그림

| 주차     | 목표                 | 마감 시점 결과물                               |
| ------ | ------------------ | --------------------------------------- |
| Week 1 | 신원/역할 + DB 기반      | 사업자 인증된 종합/전문 회원이 로그인해 회사 프로필을 가질 수 있다  |
| Week 2 | 입찰공고 시스템           | 종합사가 공고를 등록·게시하고, 전문사가 검색·열람할 수 있다      |
| Week 3 | 입찰 제출 → 개찰 → 낙찰    | 전문사가 입찰을 내고, 마감 후 종합사가 개찰·낙찰자를 선정할 수 있다 |
| Week 4 | 알림 / 관리자 / 보안 / 배포 | 알림이 가고, 관리자 운영이 가능하며, 운영 환경에 안정 배포된다    |

## ⚙️ Day 0 — 시작 전 준비 (반드시 먼저)

오늘 안에 신청만 걸어두면 평행 진행됩니다.

- [ ] **국세청 사업자등록 진위확인 API** 신청 (data.go.kr 공공데이터포털, 1~3일 승인)
- [ ] **KISCON 건설업등록 정보 조회** 방법 확인 (API 미제공 → CSV 다운로드 또는 수동 검증 흐름)
- [ ] **PG사** 가입 신청 — 토스페이먼츠 또는 포트원 (1~2주 심사)
- [ ] **본인인증** 사업자 신청 — KCP 또는 NICE (1~2주)

- [ ] **GitHub 저장소** — `dev`, `main` 브랜치 분리 + Vercel 자동 배포 환경 분리(Preview/Production)
- [ ] **DB** — Supabase/Neon 운영용 인스턴스 분리 (현재 DB와 별도)

# 🟦 Week 1 — 신원·역할 체계 + DB 기반

### Day 1 (월) — DB 스키마 설계

- [ ] ERD 작성: `users`, 
  companies`, `company_verifications`, `licenses`, `terms_consents`
- [ ] 회원 유형 enum: `GENERAL` (종합) / `SPECIALTY` (전문) / `ADMIN`
- [ ] Prisma 스키마 작성 → 마이그레이션 1차
- [ ] 운영 DB와 개발 DB 분리 확인

- **Done**: 마이그레이션 통과, 시드 스크립트로 관리자 1명 생성됨

### Day 2 (화) — 가입 플로우 분기

- [ ] 가입 시 회원 유형 선택 화면
- [ ] 종합/전문에 따라 다른 필드 노출 (전문은 면허 종목 다중 선택)
- [ ] 가입 후 상태 = `PENDING` (서류 미제출)
- [ ] 약관·개인정보 동의 체크박스 + 동의 시각 저장

- **Done**: 신규 가입자가 유형별로 분리되어 DB에 저장됨

### Day 3 (수) — 사업자 인증 ①

- [ ] 사업자등록증 업로드 (이미지/PDF, 10MB 이하)
- [ ] 국세청 사업자 진위확인 API 연동 → 사업자번호 + 대표자명 + 개업일 검증
- [ ] 검증 성공 시 `business_verified = true`

- **Done**: 잘못된 사업자번호는 가입 진행 차단됨

### Day 4 (목) — 사업자 인증 ② + 관리자 승인 큐

- [ ] 건설업등록증 업로드 + 면허번호·종목·등록일 입력
- [ ] 관리자 페이지 임시 페이지: **승인 대기 회원 목록** + 첨부 미리보기 + 승인/반려 버튼
- [ ] 반려 시 사유 입력 + 이메일 발송

- **Done**: 관리자가 1명을 승인하면 그 회원이 로그인 후 정상 사용 가능

### Day 5 (금) — 회사 프로필

- [ ] 회사 프로필 조회/수정 페이지
- [ ] 항목: 상호, 대표자, 주소, 연락처, 면허종목, 시공능력평가액, 주력 지역, 시공실적, 보유장비/인력
- [ ] 프로필 사진/로고 업로드

- **Done**: 본인이 자기 회사 정보를 보고 수정할 수 있음

### Day 6 (토) — 권한·미들웨어

- [ ] 라우트 가드: `미인증 / 종합 / 전문 / 관리자` 접근 제어
- [ ] 비밀번호 재설정 (이메일 토큰)
- [ ] 비밀번호 변경

- **Done**: 권한 없는 사용자는 보호 페이지에 접근 시 자동 리다이렉트

### Day 7 (일) — Week 1 마감

- [ ] 통합 테스트 시나리오: 가입→사업자 인증→관리자 승인→로그인→프로필 작성
- [ ] Sentry/로그 점검
- [ ] Preview 환경에 배포 후 손으로 직접 시나리오 돌려보기

- **Done**: 가입~프로필까지 끊김 없이 동작하는 영상/스크린샷 확보

# 🟩 Week 2 — 입찰공고 시스템

### Day 8 (월) — 공고 데이터 모델

- [ ] 공종 분류 마스터 테이블: 대분류(토목/건축/전기/설비/통신/조경 등) → 중분류
- [ ] 시드 데이터로 분류 코드 입력
- [ ] `bid_notices`, `bid_notice_attachments`, `bid_notice_categories` 테이블

- **Done**: 공종 분류 드롭다운에 데이터가 뜸

### Day 9 (화) — 공고 작성 폼 (기본)

- [ ] 종합사 전용 라우트 `/notices/new`
- [ ] 입력: 제목, 본문(WYSIWYG 또는 마크다운), 공종, 지역(시·도/시·군·구), 공사기간, 추정금액, 마감일시, 개찰일시
- [ ] 임시저장 / 등록 버튼 (등록 시 `상태 = 공고중`)

- **Done**: 폼 제출하면 DB에 공고가 저장됨

### Day 10 (수) — 첨부파일 업로드

- [ ] S3/Supabase Storage **Pre-signed URL** 발급 API
- [ ] 다중 파일 업로드 (도면 PDF/DWG, 시방서, 산출내역서)
- [ ] 파일별 권한 표시 (공개/자격충족자만)
- [ ] 진행률 표시 + 큰 파일은 multipart upload

- **Done**: 100MB 도면이 업로드되어 DB에 메타데이터가 남음

### Day 11 (목) — 자격조건·입찰방식·임시저장

- [ ] 자격조건: 면허 종목, 시공능력평가액 하한, 지역제한, 실적조건
- [ ] 입찰방식: 공개 / 지명 (지명은 초대할 회사 선택)
- [ ] 입찰보증금/이행보증 표기
- [ ] 임시저장 → 다시 들어가서 이어 작성

- **Done**: 자격 미달 전문사는 추후 입찰 단계에서 차단

### Day 12 (금) — 공고 목록 + 검색·필터

- [ ] `/notices` 목록 페이지
- [ ] 정렬: 마감임박 / 최신 / 추정금액
- [ ] 필터: 공종, 지역, 금액 범위, 키워드
- [ ] 페이지네이션 또는 무한 스크롤
- [ ] 검색은 일단 PostgreSQL `ILIKE` + 인덱스, 나중에 Meilisearch로 교체

- **Done**: 전문사가 공고를 검색해 결과를 받아볼 수 있음

### Day 13 (토) — 공고 상세 + 권한별 첨부 다운로드

- [ ] `/notices/[id]` 상세 페이지
- [ ] 자격조건 충족 여부 자동 판정 → 충족 시에만 다운로드 버튼 활성화
- [ ] Pre-signed URL은 5분 만료
- [ ] 미인증/미로그인 사용자는 마스킹 처리

- **Done**: 자격 미달 회원은 도면을 못 받음

### Day 14 (일) — 공고 상태 + 자동 마감 + Week 2 마감

- [ ] 공고 상태 머신: 작성중 → 공고중 → 마감 → 개찰 → 낙찰/유찰
- [ ] **마감 시각 도달 시 자동 전이** (Vercel Cron 또는 DB 스케줄)
- [ ] 종합사 본인 공고 수정/취소 (입찰 1건이라도 들어오면 일부 필드 잠금)
- [ ] 통합 시나리오 점검 + Preview 배포

- **Done**: 마감 시각 1분 후 상태가 자동으로 `마감`으로 바뀜

# 🟧 Week 3 — 입찰 제출 → 개찰 → 낙찰

### Day 15 (월) — 입찰서 작성 폼

- [ ] 전문사 전용 `/notices/[id]/bid`
- [ ] 자격조건 미달 시 차단
- [ ] 입력: 입찰금액, 공기, 비고, 산출내역서 첨부, 자격증빙 첨부
- [ ] 다른 사람의 입찰가는 절대 안 보임 (UI 차단 + 서버 차단)

- **Done**: 입찰서 작성 폼 완성

### Day 16 (화) — 입찰서 제출 + 암호화 저장

- [ ] 입찰금액은 **DB에 암호화 저장** (KMS 또는 앱 레벨 AES-256)
- [ ] 마감 전에는 본인만 조회·수정 가능
- [ ] **종합사도 마감 전에는 입찰가 못 봄** (서버 측 권한 차단)
- [ ] 감사 로그(audit log)에 제출 시각·IP 기록

- **Done**: DB를 직접 봐도 입찰가가 평문으로 안 보임

### Day 17 (수) — 입찰 수정/철회 + 마감 잠금

- [ ] 마감 시각 전에는 수정·철회 가능
- [ ] **마감 시각 이후 모든 변경 차단** (서버 시간 기준, 클라이언트 시계 신뢰 X)
- [ ] 입찰 이력 페이지 (전문사: 내 입찰 목록 / 종합사: 내 공고에 들어온 입찰 — 단, 마감 전엔 참여사 수만)

- **Done**: 마감 1초 후 수정 시도하면 거절됨

### Day 18 (목) — 자동 개찰 + 개찰 화면

- [ ] 개찰 시각 도달 시 입찰가 복호화 권한 부여 (종합사 본인에게만)
- [ ] `/notices/[id]/opening` 화면: 참여사, 입찰금액, 공기, 첨부 다운로드
- [ ] 정렬: 입찰가 오름차순 / 평가점수
- [ ] 모든 참여사에게 "개찰됨" 알림 (이메일은 다음 주에 붙여도 OK)

- **Done**: 종합사가 개찰 화면에서 모든 입찰가를 볼 수 있음

### Day 19 (금) — 낙찰자 선정

- [ ] 낙찰자 1명 선택 → 상태 `낙찰`
- [ ] 다른 참여사는 자동으로 `탈락`
- [ ] 낙찰 사유 입력(선택)
- [ ] 유찰 처리(참여 0 또는 자격 미달 0건일 때 자동)
- [ ] 낙찰 결과 화면 (전문사 본인 결과만)

- **Done**: 낙찰자 1명 + 탈락자 N명 상태가 정상 분기

### Day 20 (토) — 공고별 Q&A

- [ ] `/notices/[id]/qna` — 질문(전문사) → 답변(종합사)
- [ ] 답변은 모든 참여사에게 공개(공정성)
- [ ] 익명 옵션(질문 작성자 표시 여부)
- [ ] 마감 전까지만 질문 가능

- **Done**: 질문/답변이 정상 등록·표시됨

### Day 21 (일) — Week 3 시나리오 점검

- [ ] 종단 간 시나리오 직접 실행: 공고 등록 → 전문사 3명 입찰 → 마감 → 개찰 → 낙찰 → 결과 확인
- [ ] 시간 동기화·암호화·권한 침범 시도(다른 종합사가 남의 공고 개찰 시도) 모두 거절되는지 확인
- [ ] 버그 수정 + Preview 배포

- **Done**: 풀 플로우 1회 무결점 통과 영상 확보
